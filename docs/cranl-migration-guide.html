Migration Guide

Move Esports Community Bot from UGREEN NAS to Cranl

A practical guide for moving the bot and dashboard from the current SQLite-backed NAS deployment to Cranl with managed PostgreSQL, without breaking Discord messages, predictions, auth sessions, or Liquipedia rate safety.

1. Summary

The safest approach is to treat this as two separate moves: first migrate the data into PostgreSQL, then port the app runtime from synchronous SQLite to async PostgreSQL.

Do not start the Cranl bot as production yet. The current runtime still imports better-sqlite3, so it would use or create a SQLite file instead of the Cranl PostgreSQL database.

The importer and PostgreSQL schema are prepared for rehearsals. Better Auth can now resolve PostgreSQL through DATABASE_URL, but the app-owned bot/dashboard tables still need the remaining database-layer port before the bot can run live on Cranl.

2. Cranl Database Values

Use These

  • Type: PostgreSQL
  • Region: MENA / Saudi Arabia
  • Display name: esports-community-postgres
  • Database name: esports_community
  • Username: ecb_app

Keep Private

  • PostgreSQL password
  • Internal and external connection URLs
  • Discord bot token
  • BETTER_AUTH_SECRET
  • R2/object-storage secrets

Use Cranl's internal DATABASE_URL for the deployed app. Use the external URL only from your local machine for migration and testing.

3. Migration Phases

Prepare Cranl PostgreSQL

Create the database, keep it idle, and do not run the bot from Cranl yet.

Rehearse Data Import

Use a copied SQLite backup and import into a disposable or empty PostgreSQL target.

Port Runtime DB Layer

Replace synchronous SQLite calls with async PostgreSQL queries behind the existing modules.

Test Cranl With Bot Disabled

Run RUN_BOT=false and RUN_WEB=true first to test dashboard/auth safely.

Final Cutover

Stop NAS, take final backup, import final data, then start Cranl as the only live bot.

4. NAS Backup

Stop the NAS bot before the final backup so SQLite and its WAL file are consistent. 

cd /volume1/docker/ECB
docker compose stop esports-community-bot
sqlite3 ./data/bot.sqlite ".backup './backups/bot-$(date +%Y%m%d-%H%M%S).sqlite'"

If sqlite3 is not installed on the NAS, stop the container and copy all three files:

  • bot.sqlite
  • bot.sqlite-wal
  • bot.sqlite-shm

5. Import Into PostgreSQL

Start with a dry run. This only reads the SQLite file and prints row counts. 

npm run db:sqlite-to-pg -- --dry-run --sqlite C:\path\to\bot.sqlite

Then import into PostgreSQL by setting DATABASE_URL in your shell. Do not save the real password into Git. 

$env:DATABASE_URL = "postgresql://ecb_app:<password>@<external-host>:<external-port>/esports_community"
$env:PGSSLMODE = "require"
npm run db:sqlite-to-pg -- --sqlite C:\path\to\bot.sqlite
Remove-Item Env:\DATABASE_URL
Remove-Item Env:\PGSSLMODE
The importer is safe to rehearse: it uses a whitelist of tables, preserves IDs, resets identity sequences, and uses ON CONFLICT DO NOTHING so reruns do not overwrite rows.

6. Cranl Environment

Keep the same secrets from the NAS deployment. The most important value is BETTER_AUTH_SECRET; changing it can break existing dashboard sessions and linked Discord account data. 

DATABASE_URL=postgresql://...
DB_DRIVER=postgres
NODE_ENV=production
RUN_BOT=false
RUN_WEB=true
PORT=3000
WEB_PORT=3000
WEB_HOST=0.0.0.0
BETTER_AUTH_SECRET=<same value as NAS>
BETTER_AUTH_URL=https://<cranl-or-custom-domain>
EWC_DASHBOARD_PUBLIC_URL=https://<cranl-or-custom-domain>
EWC_DASHBOARD_INTERNAL_URL=http://localhost:3000
DISCORD_TOKEN=<same bot token>
DISCORD_CLIENT_ID=<app id>
DISCORD_CLIENT_SECRET=<client secret>
DISCORD_GUILD_ID=1087350030693838918

Begin with RUN_BOT=false on Cranl. Turn it on only during final cutover after the NAS bot is stopped.

7. Testing Checklist

  • Run the SQLite import dry run and confirm table counts look right.
  • Import into a disposable PostgreSQL database and confirm target counts match.
  • Deploy Cranl with RUN_BOT=false and verify the dashboard starts.
  • Log in with Discord and confirm Better Auth still works.
  • Open admin pages and confirm games, news, audit, predictions, and leaderboard pages load.
  • Submit a test prediction on a staging copy if possible.
  • Confirm Liquipedia rate-limit/backoff state will persist across Cranl restarts.
  • Confirm the NAS bot is the only live Discord bot until cutover.

8. Final Cutover

  1. Announce a short maintenance window.
  2. Stop the NAS bot.
  3. Take the final SQLite backup.
  4. Run the final import into Cranl PostgreSQL.
  5. Deploy Cranl with RUN_BOT=true and RUN_WEB=true.
  6. Watch logs for one bot login and no duplicate process.
  7. Run a Discord smoke test: one public command, one admin command, one dashboard action.
Once Cranl is healthy, keep the NAS deployment stopped for a while as rollback insurance. Do not delete the NAS data immediately.

9. Rollback

If Cranl fails during cutover, stop the Cranl app first, then start the NAS bot again. This avoids duplicate Discord bot sessions. 

# Cranl dashboard
Stop application

# NAS
cd /volume1/docker/ECB
docker compose up -d

Keep the failed Cranl PostgreSQL database intact until we inspect it. It may contain useful import logs or partial state from the failed attempt.
